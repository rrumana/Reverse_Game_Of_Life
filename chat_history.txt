C. Can your current architecture help with SAT → Rev-GOL?
Yes. Very much.

The important distinction is this:

offline gadget discovery may use SAT search, hill climbing, genetic search, brute force, anything;

online reduction from SAT instance to Rev-GOL instance must use only a fixed, previously discovered gadget library and assemble it in polynomial time.

That is standard and perfectly legitimate.

And there is a nice precedent: Salo and Törmä explicitly say their Life gadgets were found by SAT-based search, and Section 9 of their paper describes a search framework whose goal is to find a pattern 
P
P whose preimages restricted to a domain 
D
D realize exactly a prescribed set 
F
F of allowed local behaviors. That is extremely close to what you would want for a gadget synthesizer. 

They also describe a usable gadget basis: wires, a phase-shifting inverter, a wire crossing, and a universal logical gate; by composing with inverters they obtain AND/OR-style behavior. Their composite gate tiles are then assembled as fixed rectangular Life patterns with connector positions and zero boundary elsewhere. 

So the clean program plan is:

Offline phase: gadget synthesis and verification
Use your existing Rev-GOL→SAT engine as an oracle.

Pick a bounded rectangle for a candidate gadget.

Define interface “ports” on the boundary.

Define allowed port behaviors in the predecessor.

Search for a target pattern whose preimages realize exactly those allowed behaviors.

Prove the gadget correct by exhaustive SAT/UNSAT checks over all port assignments.

This is where your current architecture is already useful:
ConstraintGenerator + forced-cell constraints + SolutionValidator is basically the nucleus of a gadget search engine. 

 


Online phase: actual SAT-instance compiler
For the formal NP-hardness reduction, I would reduce from Circuit-SAT, not directly from 3SAT.

That makes the construction much cleaner:

Parse a SAT instance.

Tseitin-transform it into a small Boolean circuit / DAG.

Lay out the circuit on a rectangular grid.

Replace each wire, turn, crossover, inverter, and gate by the corresponding fixed Rev-GOL gadget.

Force the output wire to be true.

Emit the resulting Life target pattern.

The size is polynomial because every gadget has constant size and routing growth is polynomial in the circuit size.

A practical module plan for a SAT → Rev-GOL compiler
A good Rust split would be:

gadget_spec.rs
Stores a fixed gadget’s target cells, port locations, port protocol, and semantics.

gadget_search.rs
Offline SAT-assisted search for candidate gadgets.

gadget_verify.rs
Exhaustive proof checker: for each port assignment, SAT if allowed, UNSAT if forbidden.

circuit.rs
Boolean circuit IR: inputs, NOT, OR/AND, splitters.

router.rs
Orthogonal placement and routing with crossings.

compiler.rs
Turns a circuit into one target Rev-GOL instance using the gadget library.

emit.rs
Outputs plaintext / RLE / figure coordinates.

proof_data.rs
Stores the explicit verified gadget patterns and the local lemmas you need in the paper.
